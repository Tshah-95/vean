# corpus

Real `.mlt` files that exercise the Move-0 gates. The corpus is the oracle for
two things:

1. **Round-trip** — `parse(.mlt) → IR → serialize → .mlt` must be semantically
   equal to the input (byte-identical for files vean itself emits).
2. **Render-faithfulness** — `melt` rendering the re-emitted XML must produce
   still frames that match rendering the original (hash/SSIM within tolerance).

## What goes here

Keep a spread that covers the format's real shapes:

- **hand-authored** — minimal, legible files for specific features (a single
  clip; a gap via `<blank>`; a cross-dissolve via nested tractor; a keyframed
  filter; multi-track with a transition on the field; an audio track + gain).
- **vean-emitted** — files our own serializer produced (these must round-trip
  byte-identically).
- **Shotcut-saved** — files saved out of Shotcut, to prove we parse real-world
  output (with its `shotcut:` namespace, proxies, `time_format="clock"`, etc.).

Source clips referenced by corpus files should be tiny and either generated
(color/test-pattern producers) or committed alongside if small. Rendered outputs
are **not** committed — they go to `out/` (gitignored) and are regenerated by the
verification run.
